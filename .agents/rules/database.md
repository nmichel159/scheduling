# Antigravity Rule: Database Configuration, Migrations & Transactions

This document specifies the strict setup standards, connection rules, migration protocols, and transactional boundaries for the application's relational database layer.

---

## 1. Database Connection & Session Management
* **Unified Session Injection:** The database session lifecycles must be managed strictly using a single, unified generator function `get_db()` inside `backend/database.py` (or `backend/db/session.py`).
* **Context Isolation:** Every API request must get its own isolated database session via FastAPI's `Depends(get_db)`. The session must be automatically closed (`db.close()`) immediately after the request lifecycle ends.
* **Declarative Base:** All SQLAlchemy models must import and inherit from the exact same `Base = declarative_base()` instance to ensure they are tracked together by the ORM metadata layer.

---

## 2. Alembic Migration Protocol
When changes are made to SQLAlchemy models, database migrations must be explicitly generated using Alembic. 
* **No Direct DB Mutations:** Modifying production or local schema states manually or via `Base.metadata.create_all(bind=engine)` in production code is strictly forbidden.
* **Migration Ordering:** Every schema change must have a unique, autogenerated migration script under `backend/migrations/versions/` (or `backend/database/versions/`).
* **Migration Metadata Tracking:** The `env.py` file inside the Alembic directory must explicitly import all application models (e.g., `from backend.models.employee import Employee`, etc.) to make sure `target_metadata = Base.metadata` can accurately detect all structural discrepancies during `alembic revision --autogenerate`.

---
